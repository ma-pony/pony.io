## 0. 总结论

这两篇可以合成一条很强的 AI Agent 学习主线：

> 自动化科研解决“Agent 如何完成复杂研究任务”；持续学习解决“Agent 如何从历史任务中持续变强”。

对老板当前目标来说，它们的价值不是读论文本身，而是把 AI Agent 求职叙事从“会调 API / 会做 RAG demo”升级为：

> 能设计一个长期运行、能记忆、能评估、能复盘、能逐步改进的工作型 Agent 系统。

最适合作品集转化的方向是：

> Self-Improving Research Agent：一个能持续监控论文 / GitHub / 行业信号，生成研究简报，记录反馈，并改进检索、筛选和写作策略的 Agent。

## 1. 两篇论文各自解决什么问题

### 1.1 持续学习：解决“模型和 Agent 如何更新”

核心矛盾：LLM 往往是静态训练、静态部署，但世界、用户偏好、任务分布一直变化。

这篇把持续学习和自我迭代统一到三个问题：

- 学什么：knowledge、skills、alignment、reasoning。
- 怎么学：外部监督、自生成信号、结构适配、replay、regularization、parameter isolation、self-training、test-time adaptation。
- 什么时候学：offline、periodic、online、event-triggered、test-time。

它的底层矛盾是 `stability-plasticity dilemma`：

- plasticity：系统要能吸收新知识和新技能。
- stability：系统不能因为学新东西而忘掉旧能力、旧约束和旧偏好。

对应用工程的启发：

- 企业 Agent 里的“记忆”只是第一步，更难的是“记忆如何被验证、更新、压缩、遗忘和回滚”。
- RAG 可以解决实时事实更新，但不等于模型学会了新技能。
- 真正可持续优化的 Agent 需要 eval，否则无法判断一次更新是变好还是变坏。
- 自我改进必须有外部验证信号，否则容易 reward hacking、模型坍缩或策略越跑越偏。

### 1.2 自动化科研：解决“Agent 如何完成复杂任务”

核心变化：AI 从 copilot 变成 colleague，从辅助写作、辅助代码，走向能独立执行研究流程的 Agent。

这篇给了一个 L1-L5 自治等级框架：

| 等级 | 含义 | 例子 | 人的角色 |
|---|---|---|---|
| L1 | 自动补全 | Copilot / TabNine | 人驱动每一步 |
| L2 | 任务执行 | ChatGPT + tools / Claude chat | 人指定任务并批准动作 |
| L3 | 多步骤执行 + 检查点 | Claude Code / Cursor Agent | 人设目标，在检查点审核 |
| L4 | 有边界的全自治 | Devin / AI Scientist / SWE-Agent | 人给目标，最后验收 |
| L5 | 自主研究议程 | 假想系统 | 人给研究方向，Agent 自选问题 |

这篇认为当前前沿系统大多在 L4，L5 还没有真正实现。关键瓶颈不是模型不会生成文本，而是：

- persistent knowledge accumulation：长期知识积累。
- reliable self-evaluation：可靠自评。
- principled scaling of agent architectures：可原则化扩展的 Agent 架构。

对应用工程的启发：

- 高阶 Agent 不是聊天机器人，而是“目标 + 状态 + 工具 + 反馈 + 评测 + 退出条件”的系统。
- 多 Agent 不一定高级，只有在角色分工、并行探索、互相审查明确时才有价值。
- 生产系统更常见的是 workflow + 局部 agent，而不是完全自由 agent。
- 可靠 Agent 的关键不是一次跑通，而是能处理失败、循环、成本、上下文污染和不可复现。

## 2. 两篇论文的交集：自我改进 Agent

两篇合起来，最值得抓的不是“自动化科研”这个窄场景，而是：

> Agent 如何从长期任务历史中学习，逐步改进自己的策略、工具调用和判断标准。

可以拆成 5 个模块：

1. 任务执行：规划、检索、实验、写作、工具调用。
2. 过程记录：保存每次任务的输入、动作、结果、失败原因和人工反馈。
3. 经验抽取：从成功和失败轨迹中抽取可复用规则。
4. 策略更新：调整 prompt、检索策略、工具选择、任务分解方式和审核标准。
5. 回归评测：用固定 golden set 判断更新是否提升质量，避免退化。

这就是适合求职作品集的核心表达：

> 我不只是做一个能回答问题的 Agent，而是做一个有运行记录、有反馈闭环、有评测指标、能持续改进的 Agent 系统。

## 3. 关键概念速记

### 3.1 Catastrophic Forgetting

模型学新任务时，旧任务表现突然下降。

工程类比：Agent 更新 prompt / 记忆 / 策略后，新任务表现变好，但旧场景变差。比如日报写得更深了，但格式稳定性下降；检索更激进了，但误召回增加。

面试表达：

> 持续优化系统时，我会把每次策略变更都接入回归集，既看新任务收益，也看旧任务退化。没有 regression eval 的“自我改进”很容易只是局部变好、全局变差。

### 3.2 Stability-Plasticity

系统既要能学新东西，又要保持旧能力稳定。

工程类比：

- 太稳定：系统无法吸收用户反馈，永远重复旧错误。
- 太可塑：系统每次反馈都改过头，输出风格和行为漂移。

作品集里可以做成指标：

- 新任务成功率。
- 历史任务回归通过率。
- 人工反馈采纳率。
- 失败模式复现率。

### 3.3 Self-Improvement Collapse

如果系统只用自己的输出训练自己，没有外部真实数据或验证信号，可能逐步坍缩到低质量固定点。

工程启发：

- Agent 反思不能只靠“自己觉得自己对”。
- 必须引入外部信号：测试、真实用户反馈、引用材料、人工审核、可执行代码、benchmark。

### 3.4 Cognitive Loop Trap

Agent 卡在重复无效策略中：重复同一命令、换皮尝试同一方案、复杂化但没有进展。

工程防护：

- 记录已尝试动作和失败原因。
- 设置 step limit / cost limit / time limit。
- 连续失败后强制切换策略。
- 让 reviewer / critic 检测循环。
- context 压缩时保留“失败过什么”，而不是只保留最终目标。

### 3.5 Persistent Knowledge Across Sessions

长上下文不等于记忆。研究型 Agent 需要跨 session 记住：

- 读过哪些论文。
- 哪些结论有证据。
- 哪些假设被推翻。
- 哪些方法适合当前领域。
- 哪些反馈已经被吸收。

更好的结构不是单纯 vector store，而是：

- episodic memory：一次任务的轨迹和结果。
- semantic memory：抽象后的事实、规则和判断。
- knowledge graph：论文、概念、证据、反驳、时间关系。
- eval memory：哪些修改提升或降低了指标。

## 4. 对老板求职最有用的面试回答

### Q1：Agent 和普通 Workflow 有什么区别？

Workflow 是预定义流程，稳定、可控，适合步骤清楚的任务。Agent 是根据状态动态选择下一步，适合开放任务、多工具任务和需要探索的任务。生产系统通常不是完全自由 Agent，而是 workflow 负责主干，Agent 负责局部不确定环节，比如检索、分析、诊断和修复。

### Q2：为什么自动化科研 Agent 难？

难点不在生成一篇文章，而在长期可靠地完成研究循环：选题、文献、假设、实验、验证、写作、复现和自评。当前系统最大瓶颈是长期记忆、可靠评估、成本控制、可复现性，以及避免 Agent 陷入循环。

### Q3：为什么持续学习对 Agent 重要？

Agent 如果每次 session 都从零开始，就会重复错误，无法积累领域知识。持续学习让 Agent 能从历史任务中抽取经验，改进策略和工具使用。但这必须配合评测和回滚，否则会出现遗忘、行为漂移或自我强化错误。

### Q4：如何设计一个会自我改进但不失控的 Agent？

我会把自我改进限制在可审计层：先更新 memory、prompt、retrieval policy 和 tool routing，而不是直接更新模型参数。每次更新都经过 golden set 回归评测，保留版本记录和回滚机制。涉及发布、交易、删除数据等高风险动作，保持 human-in-the-loop。

### Q5：长上下文、RAG、Memory 的区别是什么？

长上下文是一次推理能看到更多内容；RAG 是从外部知识库取相关材料；Memory 是跨任务写入、检索、更新、遗忘和权限管理的系统。一个长期运行的 Agent 不能只靠长上下文，它需要结构化记忆和反馈闭环。

## 5. 可以转成作品集的项目：Self-Improving Research Agent

实施文档：[Self-Improving Research Agent 作品集实施计划](/workspace/self-improving-research-agent-plan)

评测集：[Self-Improving Research Agent Golden Set](/workspace/self-improving-research-agent-golden-set)

### 5.1 项目定位

目标岗位：AI Agent 应用工程、RAG 工程、Agent Workflow、AI 自动化、研究工具方向。

业务场景：持续监控 AI Agent / LLM / RAG / Eval 方向论文、GitHub 项目和行业信号，生成日报 / 研究笔记 / 社媒素材，并根据人工反馈逐步优化筛选和写作策略。

一句话：

> 做一个能长期运行的研究型 Agent：不只会搜资料和写摘要，还能记住过去的判断、吸收反馈、避免重复错误，并用评测集验证自己是否真的变好了。

### 5.2 MVP 模块

1. Source Collector
   - 输入：arXiv、GitHub Trending、Hacker News、博客、X 关键词。
   - 输出：候选项目 / 论文 / 新闻。

2. Research Planner
   - 根据主题生成分析计划。
   - 判断需要读论文、看 repo、跑 demo、还是只做信息记录。

3. RAG / Evidence Layer
   - 保存原文、README、论文片段、关键引用。
   - 回答必须带来源。

4. Memory Layer
   - 记录老板偏好、历史选题、已经判断过的项目、失败案例。
   - 区分 episodic memory 和 semantic memory。

5. Feedback Loop
   - 人工标记：有价值 / 无价值 / 太空泛 / 可发推 / 可做博客 / 可做项目。
   - 抽取反馈规则，例如“不要只写行业判断，要给读者具体动作”。

6. Eval Layer
   - 固定 20 条历史样本做 golden set。
   - 指标：相关性、事实准确、引用质量、可执行建议、重复率、人工采用率。

7. Publishing / Export
   - 输出 Obsidian 笔记、日报、X 草稿、博客草稿。
   - 所有公开发布保留人工审核。

### 5.3 技术栈建议

- Python / TypeScript 均可。
- LangGraph 或轻量自写 workflow。
- SQLite / Postgres 存任务记录。
- 向量库可先用 Chroma / LanceDB / pgvector。
- BM25 + embedding hybrid search。
- LLM-as-judge + 人工抽检。
- Obsidian 作为可见知识库。

### 5.4 评测指标

| 指标 | 意义 |
|---|---|
| source precision | 采集内容里真正相关的比例 |
| citation accuracy | 引用是否支持结论 |
| novelty score | 是否避免重复旧内容 |
| actionability | 是否给出可执行建议 |
| human adoption rate | 人工采用为日报 / 推文 / 博客的比例 |
| regression pass rate | 策略更新后旧样本是否退化 |
| cost per useful item | 每条有效内容的 token / API 成本 |

### 5.5 简历 bullet 草稿

- Designed a self-improving research agent that monitors AI papers and GitHub projects, extracts evidence-grounded insights, stores persistent memory across runs, and uses regression evaluation to prevent quality degradation.
- Built an agent workflow with source collection, retrieval, memory consolidation, feedback extraction, and human-in-the-loop publishing, improving research-to-content throughput while maintaining citation grounding and reviewability.

### 5.6 面试讲解稿

这个项目不是普通的论文总结工具。我把它设计成一个长期运行的研究型 Agent：它会采集资料、做证据检索、生成分析、记录人工反馈，并把反馈转成后续可检索的经验规则。核心难点是持续学习中的 stability-plasticity：系统要能吸收新反馈，但不能因为一次修改破坏旧任务表现。所以我设计了 golden set 和回归评测，每次更新 prompt、检索策略或记忆规则后，都要验证旧样本是否退化。

## 8. 老板应该真正学会的判断框架

以后看到任何 AI Agent 项目，不要只问“它用了什么模型”，要问 7 个问题：

1. 它的自治等级是 L2、L3 还是 L4？
2. 它是否有清晰的任务边界和退出条件？
3. 它有什么工具，工具失败后怎么办？
4. 它有没有跨 session 记忆？记忆怎么写入、检索、更新和删除？
5. 它如何评估自己是否做对？
6. 它如何从历史反馈中改进？
7. 它如何防止越改越差？

这 7 个问题能直接用于：

- 看项目。
- 做作品集。
- 面试回答。
- 分析创业机会。
- 评估 Agent 产品是否靠谱。
