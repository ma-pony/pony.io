## LLM 应用基础

### Q1：为什么只会调用 API 不等于会做 AI Agent？

一句话回答：调用 API 只是把模型能力接进程序，AI Agent 还需要目标、状态、工具、决策循环、退出条件、评测和人工审核。

原理解释：普通 API 调用通常是一次输入到一次输出；Agent 是围绕任务目标，在多步过程中根据上下文和工具反馈决定下一步。两者的复杂度不在模型接口，而在任务编排、可靠性和边界控制。

工程落地：如果任务步骤清晰，应优先做 workflow；如果任务需要开放搜索、多工具调用、动态判断和状态追踪，再引入 Agent loop。无论哪种，都要有结构化输出、失败处理、日志和 eval。

项目案例：岗位机会分析 Agent 可以先做成 JD -> JSON -> 缺口分析的 workflow，再加入岗位搜索、筛选、写入 Obsidian、人工确认触达等 Agent 能力。

常见误区：把“能聊天”当成 Agent；把多调用几次 LLM 当成 Agent；没有退出条件和评测就让模型无限自主执行。

### Q2：Chatbot、workflow、agent 的区别是什么？

一句话回答：Chatbot 是对话入口，workflow 是固定流程自动化，agent 是能根据目标和反馈动态决定下一步的任务系统。

原理解释：Chatbot 解决交互问题，workflow 解决稳定流程问题，agent 解决开放任务中的动态决策问题。三者不是谁更高级，而是适用场景不同。

工程落地：企业知识库问答第一版通常是 RAG workflow，不一定需要 Agent；岗位机会分析如果只抽取 JD，也是 workflow；如果要持续搜索岗位、判断匹配度、生成触达建议并等待人工确认，才更接近 Agent。

项目案例：内容运营 Agent 的“收集素材 -> 生成草稿 -> 人工审核 -> 发布”可以先做 workflow；自动发现选题、调用多个来源、根据反馈调整风格，才逐步进入 Agent。

常见误区：为了显得先进而过度 Agent 化。实际工程里，能固定的步骤应该固定，模型自由度越大，越需要 eval 和边界。

### Q3：什么时候不需要 Agent？

一句话回答：当任务流程稳定、输入输出明确、风险高且需要强可控时，普通 workflow 往往比 Agent 更合适。

原理解释：Agent 的优势是灵活，代价是不稳定、难评测、成本更高。不是所有自动化都需要动态决策。

工程落地：批量摘要、结构化抽取、固定格式日报、简单 FAQ、确定性数据处理，都可以先做 workflow。只有当任务依赖外部环境变化、多步骤探索和工具反馈时，再考虑 Agent。

项目案例：AI 日报的初版可以是固定信息源扫描 + 摘要 + 人工确认；等有了反馈数据后，再让 Agent 自动调整信源权重和选题角度。

常见误区：把 Agent 当卖点，而不是问题的解法。面试里更好的表达是：我会先判断任务是否值得 Agent 化，再设计自治程度。
