## 0. 定位

这个项目不是再做一个“日报生成器”，而是把已有科技 / AI 日报、GitHub 项目调研、社媒草稿、Obsidian 知识库串成一个可展示的长期运行 Agent 案例。

一句话：

> 构建一个能持续监控 AI Agent / LLM / RAG / Eval 信号，生成证据化研究简报，记录人工反馈，并通过 memory + golden set 回归评测逐步改进筛选和写作策略的 Research Agent。

求职叙事：

> 我不只是会做 RAG 或调用工具，而是能设计一个长期运行、可评估、可复盘、能从历史反馈中变强的 Agent 系统。

## 1. 为什么现在做

当前总目标是 AI Agent 相关求职机会。原计划里“岗位机会分析 Agent”仍然是最直接服务求职的 P0，但这两篇 survey 暴露出一个更强的能力缺口：

- 只做岗位分析，能证明 workflow 和信息抽取。
- 做 Self-Improving Research Agent，能证明 memory、eval、自我改进、长期任务编排。

所以本项目不替代岗位机会分析 Agent，而是作为作品集的第二主线，同时反哺内容和求职：它每天产出 AI Agent 学习材料、GitHub 项目分析、X 草稿和博客素材。

## 2. MVP 边界

第一版只做“非参数化持续学习”，不训练模型，不更新权重。

只更新这些可审计层：

- source policy：去哪找资料，关键词是什么。
- ranking policy：什么内容值得进入日报 / 作品集。
- writing rubric：什么样的输出算有用。
- memory：历史判断、失败模式、用户偏好、反馈规则。
- eval cases：固定样本回归测试。

明确不做：

- 不自动发布 X/Twitter。只生成草稿，真实发布必须人工授权并走浏览器链路。
- 不做复杂多 Agent 平台。第一版用 workflow + 局部 agent，避免为了架构而架构。
- 不做模型 fine-tuning。持续学习先落在 memory、prompt、retrieval、eval 上。

## 3. 系统模块

### 3.1 Source Collector

输入源：

- GitHub Trending / GitHub search。
- arXiv / papers。
- AI 工程博客 / Hacker News / Reddit 只读。
- 已有日报和 Obsidian 笔记。

输出字段：

- title
- url
- source_type
- discovered_at
- summary_raw
- tags
- evidence_snippets
- repo_stats 或 paper_meta

### 3.2 Research Planner

根据候选内容决定分析深度：

- `record_only`：只记录，不展开。
- `daily_brief`：进入日报。
- `deep_dive`：值得拆核心代码 / 论文逻辑。
- `portfolio_candidate`：可转作品集。
- `social_draft`：适合生成 X 草稿。

判断标准：

- 是否和 AI Agent / RAG / Eval / Memory / Tool Use 强相关。
- 是否有开源代码或可验证证据。
- 是否能产出具体行动建议。
- 是否能补足求职叙事。

### 3.3 Evidence Layer

保存所有结论对应证据：

- README 关键段落。
- paper abstract / contribution / method / eval。
- GitHub stars、更新频率、核心目录。
- 代码核心模块路径。

要求：没有证据的判断只能标为“推测”，不能写成确定结论。

### 3.4 Memory Layer

区分四类记忆：

| 类型 | 内容 | 示例 |
|---|---|---|
| episodic | 某次任务轨迹和反馈 | 某项目分析过，结论是只适合日报不适合深挖 |
| semantic | 抽象后的规则 | “读者不喜欢只有行业判断，要给动作” |
| procedural | 可复用流程 | “GitHub 项目分析要看 README、issues、核心目录、examples” |
| eval memory | 策略变更和指标结果 | 调整筛选规则后，日报采用率是否提升 |

### 3.5 Feedback Loop

人工反馈标签：

- `useful`
- `too_generic`
- `not_actionable`
- `weak_evidence`
- `good_for_x`
- `good_for_blog`
- `good_for_portfolio`
- `skip_next_time`

每次反馈都要抽取一条候选规则，但规则进入长期 memory 前需要人工确认或连续出现 2 次以上。

### 3.6 Eval Layer

第一版 golden set 20 条，来源于历史日报、GitHub 项目、论文分析和社媒草稿。

评分维度：

| 指标 | 含义 |
|---|---|
| relevance | 是否和 AI Agent 求职 / 学习主线相关 |
| evidence_quality | 结论是否有引用或代码证据 |
| actionability | 是否给出读者或老板可执行动作 |
| novelty | 是否避免重复旧内容 |
| factuality | 是否有明显事实错误 |
| format_stability | 是否符合日报 / 笔记 / 草稿格式 |
| cost | 每条有效内容的时间和 token 成本 |

每次改 prompt、筛选规则或 memory 写入规则后，至少跑 5 条回归样本，避免“新任务变好，旧任务变差”。

## 4. 数据流

```text
Sources
  -> Collector
  -> Candidate Store
  -> Research Planner
  -> Evidence Extractor
  -> Brief / Note / X Draft Generator
  -> Human Review
  -> Feedback Extractor
  -> Memory Update Proposal
  -> Golden Set Regression
  -> Approved Policy / Memory Update
```

关键闭环不是“生成内容”，而是：

> 生成 -> 人工反馈 -> 抽象规则 -> 回归评测 -> 更新策略。

## 8. 面试讲解稿

这个项目不是普通日报工具。我把它设计成一个长期运行的研究型 Agent：它会采集 AI 论文和开源项目，抽取证据，生成研究简报和社媒草稿。更关键的是，它会记录人工反馈，把反馈抽象成可复用规则，再用 golden set 做回归评测，确认策略更新没有让旧任务退化。

它背后的核心问题是 continual learning 里的 stability-plasticity：系统要能吸收新反馈，但不能因为一次调整破坏旧能力。所以我没有直接训练模型，而是先把自我改进限制在可审计层，比如 memory、prompt、检索策略和写作 rubric。高风险动作，比如公开发布，始终保留 human-in-the-loop。
