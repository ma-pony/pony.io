## 0. 为什么需要专业理论

老板当前没有 AI Agent 相关正式工作经验，如果只做几个 demo，很容易被归类为“会调 API 的应用工程师”。要更有竞争力，需要补一层专业理论能力：

> 能讲清原理、知道边界、理解关键论文、能和算法 / 平台 / 工程团队对话，并能把理论转成工程判断。

这里的理论学习不是为了走纯算法研究岗，也不是为了读博士路线，而是为了提高 AI Agent 应用工程 / RAG 应用开发 / 大模型产品落地岗位的竞争力。

目标不是“把数学全部推一遍”，而是达到：

- 面试时能解释关键概念背后的机制。
- 做方案时知道为什么这样设计。
- 遇到问题时能定位是模型、检索、上下文、工具调用、评测还是产品约束的问题。
- 和算法 / 平台团队沟通时不露怯。
- 能把专业知识转化成作品集、面试回答和职业背书内容。

## 2. 第一层：LLM 基础机制

### 必须掌握

- tokenization。
- embedding。
- Transformer 架构。
- self-attention。
- positional encoding。
- pretraining。
- instruction tuning。
- SFT。
- RLHF / RLAIF。
- DPO。
- decoding：temperature、top-p、beam search。
- context window。
- hallucination。

### 面试要能讲

#### Q1：Transformer 为什么适合语言模型？

要点：

- self-attention 能建模 token 之间的关系。
- 并行训练效率高。
- 多层 attention 可以逐步形成从局部到抽象的表示。
- 但它本质仍是基于上下文预测下一个 token，不等于真实理解或数据库查询。

#### Q2：为什么模型会幻觉？

要点：

- 训练目标是预测合理文本，不是保证事实正确。
- 当上下文不足、问题超出知识、检索材料错误或 prompt 诱导时，模型仍会生成看似合理的答案。
- 工程上要用 RAG、引用、结构化输出、拒答策略、eval 和人工审核降低风险。

#### Q3：SFT、RLHF、DPO 的区别是什么？

要点：

- SFT 是用高质量指令数据教模型按人类期望回答。
- RLHF 用奖励模型和强化学习优化偏好。
- DPO 直接用偏好数据优化模型，不显式训练奖励模型，工程上更简单。
- 对应用工程师来说，重点不是会训练，而是理解模型行为为什么受数据和偏好影响。

### 必读材料

- Attention Is All You Need。
- The Illustrated Transformer。
- InstructGPT。
- Direct Preference Optimization。

### 小实验

- 用同一个问题测试不同 temperature 下答案稳定性。
- 让模型在有上下文和无上下文时回答事实问题，对比幻觉。
- 用结构化输出约束模型，观察稳定性变化。

### 求职价值

这层让老板不会停留在“模型就是黑盒 API”。面试时能解释幻觉、上下文、temperature、SFT/RLHF/DPO 的基本机制，就比普通应用开发者强一截。

## 3. 第二层：Prompting / In-Context Learning / Reasoning

### 必须掌握

- zero-shot / few-shot。
- chain-of-thought。
- self-consistency。
- structured prompting。
- role prompting。
- prompt injection。
- instruction hierarchy。
- reasoning model 的使用边界。

### 面试要能讲

#### Q1：Prompt 工程的边界是什么？

要点：

- prompt 可以改善表达、约束格式、提供上下文和示例。
- prompt 不能解决缺数据、缺工具、缺评测、业务流程不清的问题。
- 复杂任务需要 workflow、tool calling、RAG、eval，而不是无限调 prompt。

#### Q2：为什么 few-shot 有效？

要点：

- few-shot 通过上下文示例让模型推断任务格式和隐含规则。
- 它不是训练参数，而是 in-context learning。
- 示例质量、顺序和覆盖范围会影响输出。

#### Q3：什么是 prompt injection，怎么防？

要点：

- 外部输入中包含恶意指令，试图覆盖系统指令或诱导泄露信息。
- 风险在 RAG、浏览器自动化、邮件处理、网页总结中很常见。
- 防护：分离指令和数据、工具权限控制、输出验证、敏感操作人工审核、最小权限。

### 必读材料

- Chain-of-Thought Prompting Elicits Reasoning in Large Language Models。
- Self-Consistency Improves Chain of Thought Reasoning。
- Prompt injection 相关安全文章和 OWASP LLM Top 10。

### 小实验

- 同一任务分别用 zero-shot、few-shot、structured prompt，比较稳定性。
- 给模型一段包含恶意指令的网页内容，看它是否会被 prompt injection 影响。
- 设计一个岗位 JD 抽取 prompt，要求输出 JSON schema。

### 求职价值

这层让老板能讲清“prompt 是必要但不充分的工程手段”，避免被看成只会写提示词。

## 4. 第三层：RAG 与信息检索理论

### 必须掌握

- inverted index。
- BM25。
- vector embedding。
- approximate nearest neighbor。
- dense retrieval。
- sparse retrieval。
- hybrid search。
- reranking。
- query rewriting。
- multi-hop retrieval。
- chunking。
- citation grounding。
- context precision / recall。
- faithfulness。

### 面试要能讲

#### Q1：为什么 RAG 不是简单向量库问答？

要点：

- RAG 包括文档处理、索引、召回、排序、上下文构造、生成、引用、评测、权限和更新。
- 向量检索只是其中一步。
- 生产化 RAG 的难点在召回质量、上下文污染、引用准确、权限隔离和持续评测。

#### Q2：BM25 和 embedding 检索有什么区别？

要点：

- BM25 基于关键词匹配，适合专有名词、精确词、编号、错误码。
- embedding 基于语义相似，适合同义表达、自然语言问题。
- hybrid search 结合两者，rerank 再做精排。

#### Q3：RAG 为什么仍然会幻觉？

要点：

- 检索不到正确材料。
- 检索到了但上下文太长或噪声太多。
- 模型没有忠实使用上下文。
- 问题本身需要多跳推理或外部实时信息。
- eval 不完善导致问题长期存在。

### 必读材料

- Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks。
- Dense Passage Retrieval。
- ColBERT。
- HyDE。
- RAGAS 文档或论文。

### 小实验

- 对同一文档用不同 chunk size 建索引，对比召回效果。
- 对比 BM25、embedding、hybrid search 的命中结果。
- 做 10 条 golden set，计算 context recall 和 answer faithfulness。

### 求职价值

RAG 是企业 AI Agent 最常见入口。能把 RAG 讲到检索、rerank、eval、权限和引用，比只会“接向量库”的候选人更强。

## 5. 第四层：Agent 架构与规划

### 必须掌握

- ReAct。
- Plan-and-Execute。
- Reflection / Reflexion。
- Tree of Thoughts / Graph of Thoughts。
- planner / executor / critic。
- task decomposition。
- multi-agent 的适用边界。
- agent loop。
- termination condition。
- human-in-the-loop。

### 面试要能讲

#### Q1：Agent 和 Workflow 的区别是什么？

要点：

- workflow 是预定义步骤，稳定、可控、适合流程清晰的任务。
- agent 是让模型根据状态动态决定下一步，适合开放、探索、多工具任务。
- 生产中常用的是 workflow + 局部 agent，而不是完全自由 agent。

#### Q2：ReAct 解决什么问题？

要点：

- ReAct 交替进行 reasoning 和 acting。
- 它让模型在推理过程中调用工具，并根据观察结果继续决策。
- 风险是循环、误读工具结果、错误行动，所以需要 step limit、日志、审核和 eval。

#### Q3：Multi-agent 什么时候有用？

要点：

- 有明确角色分工、并行任务、互相审查时有用。
- 如果只是为了显得高级，会增加协调成本和错误传播。
- 先单 agent / workflow 跑通，再考虑 multi-agent。

### 必读材料

- ReAct。
- Reflexion。
- Tree of Thoughts。
- Voyager。
- Generative Agents。
- SWE-agent / AgentBench / WebArena 相关材料。

### 小实验

- 用 ReAct 思路设计一个岗位信息分析流程。
- 给同一任务分别设计 workflow 版和 agent 版，比较稳定性。
- 为 agent 设计退出条件和失败处理。

### 求职价值

这层帮助老板从“做问答 demo”升级到“能设计任务系统”，这是 AI Agent 岗位的核心竞争点。

## 6. 第五层：Tool Use / Function Calling

### 必须掌握

- function calling schema。
- tool selection。
- 参数生成。
- tool result grounding。
- tool error handling。
- sandbox / permissions。
- idempotency。
- side effects。
- browser automation。

### 面试要能讲

#### Q1：Tool calling 的失败模式有哪些？

要点：

- 选错工具。
- 参数不完整或格式错误。
- 工具返回空或错误。
- 模型误读工具结果。
- 多次重复调用。
- 对有副作用操作缺少确认。
- 工具权限过大导致安全风险。

#### Q2：如何设计安全的工具调用？

要点：

- 工具 schema 清晰。
- 参数校验。
- 最小权限。
- 有副作用操作人工确认。
- 日志和回放。
- 失败重试和降级。
- 输出校验。

### 必读材料

- Toolformer。
- OpenAI / Anthropic function calling docs。
- OWASP LLM Top 10 中 tool / agent 安全部分。

### 小实验

- 设计一个岗位 JD 解析工具 schema。
- 设计一个“发送触达消息”的安全流程，必须包含人工确认。
- 记录 5 种工具调用失败 case。

### 求职价值

AI Agent 岗位很看重工具调用。能讲清副作用、安全、权限、失败处理，会显著区别于普通 demo 开发。

## 7. 第六层：Memory / Context Engineering

### 必须掌握

- short-term memory。
- long-term memory。
- episodic memory。
- semantic memory。
- memory retrieval。
- memory consolidation。
- context compression。
- recency / importance / relevance。
- privacy and forgetting。

### 面试要能讲

#### Q1：为什么长上下文不等于记忆？

要点：

- 长上下文只是一次推理能看到更多内容。
- 记忆系统需要写入、检索、更新、遗忘、权限和作用域。
- 长上下文成本高，也会引入噪声。
- 好的 memory 是把相关信息在正确时机取出来。

#### Q2：Agent memory 什么时候有价值？

要点：

- 长期用户偏好。
- 历史任务反馈。
- 项目上下文。
- 反复出现的错误和修正。
- 但不适合存短期状态、敏感信息和会过期的数据。

### 必读材料

- Generative Agents。
- MemGPT / Letta。
- Reflexion。
- Voyager。
- LangGraph / LlamaIndex memory 文档可作为工程参考。

### 小实验

- 为岗位机会分析 Agent 设计 memory schema。
- 区分哪些信息应该进 memory，哪些只留在 session。
- 比较“全塞上下文”和“按需检索记忆”的差异。

### 求职价值

Memory 是 Agent 前沿高频话题。老板自己的 Hermes 使用经验也能转化为差异化认知。

## 8. 第七层：Evaluation / Alignment

### 必须掌握

- golden set。
- case replay。
- task success rate。
- tool success rate。
- answer relevance。
- faithfulness。
- context precision / recall。
- LLM-as-judge。
- human eval。
- regression test。
- SFT / RLHF / DPO 基本逻辑。
- safety / permission / data leakage。

### 面试要能讲

#### Q1：为什么没有 eval 就没有可持续优化？

要点：

- demo 成功只是个例。
- 没有 eval 就不知道改动是变好还是变坏。
- Agent 系统有多环节：检索、规划、工具、生成、评审，每一环都需要指标。
- eval 可以支持回归测试、上线前检查和持续优化。

#### Q2：LLM-as-judge 有什么问题？

要点：

- 偏好长答案。
- 可能受格式和语气影响。
- 对事实错误不总敏感。
- 可能和业务标准不一致。
- 需要标准答案、rubric、人工抽检和多评委一致性。

#### Q3：Alignment 和应用工程有什么关系？

要点：

- 不一定要自己训练模型，但要理解模型为什么服从指令、为什么拒答、为什么偏好某类输出。
- 对外部工具、敏感数据、自动发布、交易等场景，alignment 和 safety 会直接影响产品风险。

### 必读材料

- InstructGPT。
- Constitutional AI。
- Direct Preference Optimization。
- RAGAS。
- HELM。
- SWE-bench。
- AgentBench。
- WebArena。

### 小实验

- 给 RAG 系统设计 10 条 golden set。
- 用人工 rubric 评估 5 条 agent 输出。
- 设计 task success rate 和 tool success rate。
- 比较两个 prompt / workflow 的回归结果。

### 求职价值

Eval 是从 demo 到生产的分水岭。会 eval 的应用工程师比只会搭流程的人更有竞争力。

## 9. 第八层：AI 系统工程与产品经济学

### 必须掌握

- latency。
- throughput。
- cost per task。
- token cost。
- caching。
- model routing。
- fallback 风险。
- observability。
- logging。
- privacy。
- permission。
- ROI。
- automation vs augmentation。

### 面试要能讲

#### Q1：如何判断一个任务值不值得 agent 化？

要点：

- 任务价值是否足够高。
- 是否重复发生。
- 是否需要多步骤判断。
- 数据和工具是否可获得。
- 错误成本是否可控。
- 是否能评测。
- 自动化 ROI 是否大于成本和风险。

#### Q2：如何平衡质量、延迟和成本？

要点：

- 简单任务用便宜模型或规则。
- 复杂任务用强模型。
- 检索、缓存、批处理、模型路由可以降成本。
- 高风险环节加人工审核。
- 不同场景选择不同 SLA。

### 必读材料

- LLMOps / production RAG 相关文章。
- LangSmith / LangFuse / Arize Phoenix 文档。
- OpenAI / Anthropic pricing and rate limit docs。
- 企业 AI case study。

### 小实验

- 估算一个 Agent 任务的 token 成本。
- 对比强模型和便宜模型在同一任务上的质量差异。
- 设计日志字段：input、retrieved docs、tool call、output、latency、cost、human feedback。

### 求职价值

这层让老板具备产品落地和商业判断能力，适合应用工程、解决方案和产品落地岗位。
