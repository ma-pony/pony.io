## 0. 一句话总判断

这两篇不是普通 survey，而是可以拼成一条完整的 AI Agent 能力路线：

> 自动化科研回答“Agent 如何长期执行复杂研究任务”，持续学习回答“Agent 如何从长期任务历史中变强而不退化”。

对我们最有价值的不是记住所有论文名，而是把它转成一个求职/作品集叙事：

> 我能设计一个长期运行的 Research Agent。它不只是会搜索、总结和写日报，还能记录证据、保留记忆、吸收人工反馈、做回归评测，并在可控范围内持续改进自己的筛选、检索和写作策略。

这会把能力表达从“会调 API / 会搭 RAG demo”升级到“懂 Agent 长期运行、memory、eval、feedback loop、自我改进边界和工程安全”。

## 1. 两篇文章分别在解决什么问题

### 1.1 持续学习 Survey：解决“系统如何更新”的问题

这篇的核心矛盾是：LLM 通常是一次训练、静态部署，但真实世界一直在变化。

变化包括：

- 新事实出现：论文、新闻、产品、API、政策、市场都在更新。
- 新任务出现：用户会不断提出训练时没见过的任务。
- 用户偏好变化：今天想要详细解释，明天想要简洁行动项。
- 对齐标准变化：安全、语气、价值判断会随环境改变。

作者把 continual learning 和 self-improvement 放进一个统一框架，提出三轴 taxonomy：

| 维度 | 问题 | 典型内容 |
|---|---|---|
| What | 学什么 | knowledge、skills、alignment、reasoning |
| How | 怎么学 | 外部监督、自生成信号、架构适配 |
| When | 什么时候学 | offline、periodic、online、event-triggered、test-time |

这套框架很实用，因为它能避免我们把所有“更新”混成一类。比如：

- RAG 更新索引，是更新外部知识，不是模型学会新技能。
- prompt/memory 更新，是行为策略变化，不是参数学习。
- LoRA/fine-tuning，是参数层更新，风险和收益都更大。
- online DPO/RLHF，是偏好和对齐的持续更新。

### 1.2 自动化科研 Survey：解决“系统如何自治执行复杂任务”的问题

这篇的核心变化是：AI 从 copilot 走向 colleague。

它不是讨论聊天机器人，而是讨论能执行研究循环的 agent：提出假设、查资料、设计实验、写代码、跑实验、分析结果、写报告、接受评审、再迭代。

作者提出 L1-L5 自治等级：

| 等级 | 含义 | 例子 | 人的角色 |
|---|---|---|---|
| L1 | 自动补全 | GitHub Copilot、TabNine | 人驱动每一步 |
| L2 | 任务执行 | ChatGPT + tools、Claude chat | 人指定任务并批准动作 |
| L3 | 多步骤执行 + 检查点 | Claude Code、Cursor Agent | 人设目标，在检查点审核 |
| L4 | 有边界的全自治 | Devin、AI Scientist、SWE-Agent | 人给目标，最后验收 |
| L5 | 自主研究议程 | 假想系统 | 人给研究方向，Agent 自选问题 |

关键结论是：当前前沿大多在 L4，L5 仍然是目标而非现实。瓶颈不是模型不会写字或不会调用工具，而是：

- persistent knowledge accumulation：长期知识积累。
- reliable self-evaluation：可靠自评。
- principled scaling of agent architectures：可原则化扩展的架构。

这三点正好和第一篇 continual learning 接上。

## 2. 真正的交集：Self-Improving Research Agent

两篇合起来，最值得抓的是这个问题：

> 一个长期运行的 Agent，如何从过去的任务、失败、反馈和评测中学习，改进自己的策略，同时不破坏旧能力？

这可以拆成一个闭环：

```text
任务执行
  -> 过程记录
  -> 结果评估
  -> 人工反馈
  -> 经验抽取
  -> 策略 / memory / rubric 更新
  -> golden set 回归评测
  -> 批准更新或回滚
```

这个闭环比“自动写日报”高级得多。日报只是输出物，真正的作品集价值是证明你理解：

- 为什么 Agent 会重复犯错。
- 为什么长上下文不等于长期记忆。
- 为什么自我反思不能只靠模型自嗨。
- 为什么没有 eval 的自我改进会越改越偏。
- 为什么高风险动作必须 human-in-the-loop。

## 3. 持续学习 Survey 详细分析

### 3.1 文章贡献

这篇文章主要做了四件事：

1. 把 continual learning 和 self-improvement 统一到同一套框架里。
2. 用 what / how / when 三轴梳理更新系统。
3. 系统比较五类方法：parameter isolation、regularization、replay、self-training、inference-time adaptation。
4. 提出关键开放问题：遗忘、collapse、评测、对齐、安全、Agent 集成。

我认为它最重要的价值是：它把“模型会不会持续学习”从一句空话拆成了工程问题。

### 3.2 五类方法的工程含义

| 方法族 | 核心思想 | 优点 | 代价 | 对我们项目的启发 |
|---|---|---|---|---|
| Parameter isolation | 新任务用独立 adapter / LoRA / expert | 防遗忘强 | 需要管理模块和路由 | 第一版不训练模型，但 memory/rubric 也要分域隔离 |
| Regularization | 更新时约束别偏离旧能力 | 成本相对低 | 保护有限，调参难 | 策略更新要有“别破坏旧格式/旧偏好”的约束 |
| Replay | 新数据和旧样本混合训练/评测 | 简单有效 | 需要存储和数据治理 | golden set 就是非参数版 replay |
| Self-training | 模型生成数据再训练/改进 | 可规模化 | 容易自我污染和 collapse | 生成日报后不能直接拿来训练判断，必须有人类反馈/外部证据 |
| Test-time adaptation | 推理时动态改进 | 不一定改权重 | 稳定性难保证 | Reflexion / memory / prompt replay 是第一版最现实路径 |

文章里一个很关键的判断是：没有单一方法适合所有场景，实际系统往往是 hybrid。对我们来说，第一版最合理的 hybrid 是：

- RAG / evidence layer 更新事实。
- memory 更新用户偏好和历史判断。
- prompt / rubric 更新写作策略。
- golden set 防止策略退化。
- human review 作为外部验证信号。

不要一上来 fine-tune，因为现在的目标不是训练一个模型，而是做一个可控、可解释、可展示的 Agent 系统。

### 3.3 Stability-Plasticity 是核心矛盾

stability-plasticity dilemma 可以翻译成：稳定性与可塑性的矛盾。

- plasticity：系统要能吸收新知识、新偏好、新反馈。
- stability：系统不能因为一次更新，把过去有效的能力、格式、边界和安全约束弄坏。

工程类比：

- 老板说“日报不要空泛，要给动作”，系统应该学会提高 actionability。
- 但不能因此每条新闻都硬塞行动项，导致事实分析变形。
- 老板说“X 内容要更像真人”，系统应该学会减少 AI 味。
- 但不能因此牺牲准确性，开始写模糊情绪化判断。

所以正确做法不是“收到反馈立刻改系统全局 prompt”，而是：

1. 把反馈转成候选规则。
2. 判断反馈适用范围。
3. 在 golden set 上测试是否改善。
4. 如果新规则导致旧样本退化，就缩小适用条件或回滚。

### 3.4 Self-improvement 的边界

文章强调：稳定的自我改进需要外部质量信号。

可用信号包括：

- 代码执行结果。
- 数学/逻辑 verifier。
- 检索到的事实证据。
- 人工反馈。
- benchmark / golden set。
- 多模型或多 agent 交叉审查。

危险路径是：模型生成内容 -> 模型自己评价 -> 模型根据自己评价继续改。这个闭环如果没有外部锚点，容易出现 reward hacking、输出多样性下降、模型坍缩或固定低质量 attractor。

对我们的启发很直接：

- 日报候选不能只让模型判断“是否重要”，要看来源、代码、引用、传播、和求职/作品集关系。
- X 草稿不能只让模型判断“是否好读”，要记录老板是否采纳、是否修改、是否发布。
- 作品集项目不能只生成“看起来完整”的文档，要有可运行工作流、样本、评分和回归记录。

## 4. 自动化科研 Survey 详细分析

### 4.1 文章贡献

这篇主要做了四件事：

1. 提出 L1-L5 自治等级，给 Agent 能力分级。
2. 总结四类主流架构：single-agent loops、multi-agent collaboration、hierarchical orchestration、tool-augmented execution。
3. 比较 17 个代表系统，包括 AutoGPT、BabyAGI、SWE-Agent、Devin、AI Scientist、STORM、GPT-Researcher 等。
4. 总结六大开放问题：cognitive loop、context limitation、novelty evaluation、reproducibility、safety、cost。

它最重要的价值是：把“Agent 很强”这种模糊说法变成可诊断框架。你可以问一个系统：它到底是 L2、L3 还是 L4？它的任务边界在哪里？它是否有长期记忆？它失败后如何换策略？它如何评估自己产出的研究是否真的有新意？

### 4.2 四类架构怎么选

| 架构 | 代表 | 适合场景 | 主要风险 | 对我们项目的选择 |
|---|---|---|---|---|
| Single-agent loop | ReAct、Reflexion、Self-Refine、LATS | 简单任务、明确反馈、多步执行 | 容易循环，长任务上下文爆炸 | MVP 主体可以用 workflow + 单 agent 局部分析 |
| Multi-agent | CAMEL、AutoGen、MetaGPT、debate | 需要角色分工、互审、并行探索 | 沟通噪声、成本高、角色混乱 | 初期只在审核/反驳环节用，不做复杂平台 |
| Hierarchical orchestration | supervisor-worker | 长任务、需要检查点和分解 | 信息损失、协调成本 | 适合后续扩展：collector/researcher/reviewer/publisher |
| Tool-augmented execution | SWE-Agent、Claude Code、AI Scientist | 需要代码、浏览器、检索、实验 | 工具误用和安全风险 | 必须保留工具权限边界和日志 |

文章有一个很重要的判断：架构要匹配自治等级。

- L2：简单 ReAct 足够。
- L3：需要 checkpoint、Reflexion、supervisor-worker。
- L4：需要层级编排、自我评估、错误恢复、工具沙盒。
- L5：需要长期知识、自主选题、可靠 novelty evaluation，目前还没真正实现。

我们现在不应该吹 L5。更可信的说法是：

> 第一版做 L3，部分模块具备 L4 特征：系统能自主采集、筛选、生成和回归评测，但公开发布、策略更新和高风险动作保留人工审核。

这比“全自动科研 Agent”更专业，也更不容易被面试官追问打穿。

### 4.3 六大失败模式

#### 1. Cognitive Loop Trap

Agent 会重复无效策略：重复同一命令、换皮尝试同一方案、堆复杂度但没有进展。

防护：记录已尝试动作、失败原因、step limit、cost limit、连续失败后强制换策略、reviewer 检测循环。

#### 2. Context Window Limitations

长任务会产生大量历史：文献、实验、代码、错误、决策。上下文放不下时，Agent 会忘掉早期关键事实。

防护：结构化 memory、分层摘要、检索历史、把“失败过什么”保留下来。

#### 3. Evaluation of Novel Contributions

研究质量不是普通任务完成率。新颖性、正确性、重要性很难自动评估。

防护：优先选择可验证领域；让 reviewer agent 反驳；用 portfolio-level evaluation 看一组输出的累计价值。

#### 4. Reproducibility and Determinism

Agent 输出受模型版本、温度、prompt、工具环境影响大。同一任务可能跑出不同路径。

防护：记录模型、prompt、工具版本、输入、输出、随机参数；关键样本重复跑；发布结论时区分确定事实和一次运行观察。

#### 5. Safety and Ethics

自治越高，风险越大。尤其是自动发布、自动交易、自动删除、自动执行外部动作。

防护：human-in-the-loop、权限最小化、日志、审批、动作白名单。

#### 6. Cost and Accessibility

长期 Agent 很耗 token、API、时间。成本不是小问题，会决定系统是否可长期运行。

防护：先做候选筛选，再深度分析；便宜模型做初筛，强模型做判断；缓存和去重；只对高价值样本做 deep dive。

## 7. 我对两篇文章的批判性判断

### 7.1 优点

第一，它们给了很好的语言框架。L1-L5、what/how/when、stability-plasticity、cognitive loop，这些都能直接进入面试表达。

第二，它们把 Agent 从“demo 能跑”拉到“长期系统能不能可靠运行”。这正是工程岗位更关心的点。

第三，它们很适合转作品集，因为两篇的交集天然导向 memory + eval + feedback loop。

### 7.2 局限

第一，两篇都有较强“框架化整理”的味道，真正严格实验和可复现实证有限。尤其是自动化科研部分，很多系统对比来自不同论文、不同 benchmark，不能简单横向比较。

第二，持续学习文章虽然讲了参数更新，但我们当前工程可控范围主要是非参数更新：memory、prompt、retrieval、rubric。不要把论文里的训练方法硬搬到作品集第一版。

第三，自动化科研文章对 L5 的讨论偏愿景。目前真正能落地的是 L3-L4 的有边界自治，而不是完全自主科研议程。

第四，LLM-as-judge 和 self-evaluation 仍然不可靠。我们可以用它辅助，但不能把它当唯一真相来源。

## 8. 最该记住的判断框架

以后看任何 Agent 项目，问 10 个问题：

1. 它是 L2、L3、L4 还是在假装 L5？
2. 它的任务边界和退出条件是什么？
3. 它如何选择工具？工具失败怎么办？
4. 它是否记录历史尝试和失败原因？
5. 它的 memory 是长上下文、RAG，还是真正跨 session 的结构化记忆？
6. 它如何评估输出质量？靠自评、测试、人工、还是外部 verifier？
7. 它如何从反馈中更新策略？
8. 更新后如何防止旧能力退化？
9. 高风险动作是否有 human-in-the-loop？
10. 成本是否允许它长期运行？

这 10 个问题就是评估 Agent 产品、做作品集、准备面试和判断创业机会的通用框架。

## 9. 给老板的学习路线

第一轮，不要细抠所有引用。先把主线讲顺：

- 自动化科研：L1-L5 + 四类架构 + 六大失败模式。
- 持续学习：what/how/when + stability-plasticity + replay + self-improvement collapse。

第二轮，按作品集倒推：

- memory schema 怎么设计。
- golden set 怎么设计。
- feedback 如何抽象成规则。
- 什么动作必须人工审批。

第三轮，做最小原型：

- 输入 3 个 GitHub 项目和 2 篇论文。
- 输出候选评分、证据摘要、日报段落、X 草稿。
- 老板反馈后生成规则更新建议。
- 用 5 条 golden set 测一次回归。

## 10. 最终结论

这两篇文章的真正价值，不是告诉我们“未来 AI 会自动做科研”，而是给了一个更硬的 Agent 工程判断：

> 长期 Agent 的关键不是一次性生成能力，而是持续运行中的状态、记忆、反馈、评估和更新控制。

对老板当前 AI Agent 求职/作品集主线，最值得下注的项目就是 Self-Improving Research Agent。它能同时证明：

- 你理解 RAG 和 memory 的区别。
- 你理解 Agent 自治等级和架构选择。
- 你理解 eval 是 demo 到生产的分水岭。
- 你理解自我改进不能脱离外部信号。
- 你能把理论 survey 转成可运行、可展示、可面试的工程系统。

这就是它比普通“日报生成器”更有作品集价值的地方。
