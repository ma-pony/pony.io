## 今天学什么

今天先建立 AI Agent 求职方向的第一块地基：分清 LLM 应用、Chatbot、Workflow 和 Agent 的边界。

一句话：

> LLM 应用是把大模型能力接进产品；Chatbot 是对话界面；Workflow 是预设流程自动化；Agent 是能根据目标、上下文和工具反馈动态决定下一步的任务系统。

这件事重要，是因为很多岗位写 AI Agent，但真正需要的不是“会调一次 API”，而是能把模型、工具、数据、评测、人工审核和业务流程组织成可靠系统。

## 为什么和求职相关

招聘方通常不缺会写 prompt、会接 OpenAI API 的人。更稀缺的是能回答这些问题的人：

- 这个场景到底需不需要 Agent？
- 哪些步骤应该固定成 workflow，哪些步骤才交给模型决策？
- 工具调用失败、检索失败、输出不稳定时怎么兜底？
- 如何证明系统真的变好了，而不是 demo 看起来能跑？
- 哪些动作必须 human-in-the-loop？

所以第一天的目标不是学一个新框架，而是建立判断框架。以后看项目、看岗位、做作品集，都先用这套框架拆。

## 核心概念

### 1. LLM 应用

LLM 应用是最宽的概念：只要产品里用了大模型能力，都可以算。

常见形式：

- 文本生成。
- 摘要。
- 分类。
- 信息抽取。
- 翻译。
- 问答。
- 代码生成。
- 结构化输出。

它的关键问题是：模型输出是否稳定、是否可控、是否能接进业务流程。

只做 LLM 应用，不一定是 Agent。

### 2. Chatbot

Chatbot 是一种交互形态，核心是“人问，模型答”。

它适合：

- FAQ。
- 轻量客服。
- 个人助手。
- 知识库问答入口。

但 Chatbot 的局限是：它通常不主动拆任务，不持续执行，不一定有明确的工具链和评测闭环。

所以 Chatbot 可以是 Agent 的界面，但不是 Agent 本身。

### 3. Workflow

Workflow 是预设步骤的自动化。

典型结构：

```text
输入 -> 步骤 A -> 步骤 B -> 步骤 C -> 输出
```

比如：

```text
读取岗位 JD -> 抽取结构化字段 -> 匹配能力缺口 -> 生成学习建议
```

Workflow 的优点是稳定、可控、容易评测。缺点是灵活性较弱，遇到开放问题时需要人工或模型补判断。

很多所谓 Agent 项目，本质上应该先做成 workflow。能固定的流程先固定，能减少模型自由发挥就减少。

### 4. Agent

Agent 是围绕目标行动的任务系统。它不只是回答问题，而是能根据状态和反馈选择下一步。

一个最小 Agent 通常包括：

- 目标：要完成什么任务。
- 状态：当前已经知道什么、做到哪一步。
- 工具：能搜索、读文件、调用 API、操作浏览器、写数据库等。
- 决策：下一步该调用哪个工具或输出什么。
- 记忆 / 上下文：保留必要历史。
- 退出条件：什么时候算完成，什么时候停止。
- 评测：怎么判断结果好不好。
- 人工审核：高风险动作前让人确认。

Agent 的优势是能处理开放、多步骤、动态反馈任务。代价是更不稳定、更难评测、更容易失控，工程成本更高。

## 一个小实验 / 例子

用“岗位 JD 分析”拆三种实现方式：

### 方式 A：普通 LLM 调用

输入一段 JD，让模型总结岗位要求。

优点：最快。

问题：输出格式不稳定，难以批量处理，也没有能力缺口追踪。

### 方式 B：Workflow

```text
岗位 JD -> JSON schema 抽取 -> 技能归类 -> 和个人能力表匹配 -> 输出缺口和学习建议
```

优点：稳定、可评测、适合做作品集 MVP。

问题：如果岗位来源很多、格式混乱、需要自动找岗位和持续追踪，就不够灵活。

### 方式 C：Agent

```text
目标：找到适合老板的 AI Agent 岗位并生成行动建议

循环：
1. 搜索岗位。
2. 读取 JD。
3. 抽取要求。
4. 和老板能力/作品集匹配。
5. 判断是否值得记录。
6. 生成触达或学习建议。
7. 写入 Obsidian。
8. 等待老板确认下一步。
```

这个才更接近 Agent，因为系统会根据岗位内容和匹配情况动态决定下一步。

## 面试怎么讲

面试回答模板：

> 我会把 Chatbot、Workflow 和 Agent 分开看。Chatbot 主要是对话入口；Workflow 是预设流程自动化，适合步骤清晰、可控性要求高的任务；Agent 则是围绕目标，根据上下文和工具反馈动态选择下一步的任务系统。实际工程里我不会一上来就做 Agent，而是先判断任务是否需要动态决策。如果流程能固定，我会优先 workflow；只有当任务涉及开放搜索、多工具调用、状态追踪和动态反馈时，才引入 Agent loop、退出条件、eval 和 human-in-the-loop。

再补一句体现工程判断：

> Agent 的价值不在“更像人”，而在能否比固定流程更好地完成开放任务；如果没有评测和边界控制，Agent 很容易只是一个不稳定 demo。

## 能放进作品集哪里

这篇笔记可以服务三个作品集方向：

- [AI Agent 作品集](/workspace/agent-portfolio) 里的“岗位机会分析 Agent”：先用 workflow 做 MVP，再逐步加入 Agent loop。
- 企业知识库 + 销售问答 Agent：说明为什么第一版应是 RAG workflow，而不是过度 Agent 化。
- 内容运营 Agent：说明发帖、互动等外部动作必须 human-in-the-loop。

后续作品集页面可以用这个判断框架作为“设计原则”：

> 先 workflow，后 Agent；先可控，后自治；先 eval，后扩展。
