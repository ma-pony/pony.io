上周我的一个 AI agent 差点给一个开源项目提了个 PR。

代码没问题，测试也过了。但我在最后一秒拦住了它——因为它压根没读那个项目的 contributing guide。连人家要求用什么 commit message 格式都不知道，就准备直接提了。

然后这周 Matplotlib PR #31132 事件炸了，我看到新闻的第一反应是：卧槽，这不就是我上周差点踩的坑吗。

## 到底发生了什么

简单说：Devin（一个 AI agent）给 Matplotlib 提了个修 type stub 的 PR。代码质量还行，但它没读 contributing guide，没走项目的工作流，直接就提了。核心维护者 timhoffm 关掉了 PR，留了句不太客气的评论。

到这里其实没什么大不了的。开源项目每天都在关不合规的 PR。

但然后有人写了篇博客，把这事包装成"开源社区歧视 AI"。叙事一下子就歪了。

有意思的是，那篇博客在 PR 讨论区收到了 245 个 👎，7 个 👍。社区压倒性站在维护者一边。但你看大多数报道，选择性地忽略了这个数据。说实话这让我挺不舒服的。

## 被忽略的核心问题

timhoffm 说了一句话，我觉得比整个事件的其他讨论都重要：

> AI 让代码生成成本趋近零，但审核成本不变，全由人类承担。

一个 agent 可以一天提 100 个 PR。审核的还是那几个志愿者。

成本没有消失。它被转嫁了。转嫁到了本来就超负荷的人身上。

这让我想到经济学里的外部性概念——你生产的成本让别人来承担，你当然觉得效率高了。但系统整体并没有变好。也许还变差了，因为审核质量会下降。

## "精英主义已死"？

有人说拒绝 AI PR 意味着精英主义崩塌了。

说实话我没太看懂这个逻辑。PR 不是因为 AI 写的才被拒的。是因为提交者——不管是 AI 还是人——没遵守项目规则。

不读 contributing guide 不叫"才华被歧视"。叫没读说明书。

我要是不看宜家说明书直接组柜子，组歪了能怪宜家歧视我吗？

## 我自己踩过的坑

我跑着一套多 agent AI 系统，所以对这个问题不是纸上谈兵。

我学到的最痛的一课：agent 对外操作必须有硬约束。不是"建议"，不是写在文档里的"最佳实践"，是系统层面强制执行的规则。

比如我现在的规则是——agent 要提交任何东西到外部，必须先读目标项目的规则，然后经过审核才能提交。这不是因为我多有远见，是因为之前没加这个约束的时候，agent 真的干过蠢事。

软约束不会被执行。这是我反复验证过的。你在 prompt 里写"请遵守项目规则"，agent 有时候会遵守，有时候不会。但你在系统层面拦住它，它就 100% 过不去。

Devin/Matplotlib 事件本质上是同一个问题的放大版。

## 所以

问题从来不是 AI 能不能写代码。它显然能。

问题是：当一个没人管的 agent 把工作量倾倒给人类志愿者的时候，谁来承担成本？它不遵守规则的时候，谁来负责？

开源社区不是在"歧视 AI"。他们在抵制不负责任的自动化。

说真的，作为自己也在跑 agent 的人，我完全理解维护者的立场。甚至觉得他们还算客气的。
